# Gold Layer Dimension Tables: Data Mapping and Transformation Rules

This document provides a detailed specification for populating the Dimension tables in the Gold Layer of the data warehouse. The primary source for these dimensions is the `silver.transactions` table.

## General Rules & Conventions

- **Source Table:** All dimension data (except for `Dim_Date`) is sourced from the `credit_card_project.silver.transactions` table.
- **Surrogate Keys:** Surrogate keys for each dimension are generated using the `FARM_FINGERPRINT()` function on the natural key(s) to ensure a consistent and unique 64-bit integer key.
- **Data Cleansing:** Basic data cleansing (trimming whitespace, standardizing case) is assumed to have been performed in the Silver layer. These rules focus on Gold layer transformations.
- **Uniqueness:** The `DISTINCT` keyword should be used when selecting from the source table to ensure each dimension record is unique based on its natural key.

## 1. `dim_customer`

This dimension stores customer information and is managed as a Slowly Changing Dimension (SCD) Type 2 to track changes in customer attributes over time.

- **Target Table:** `credit_card_project.gold.dim_customer`
- **Source:** `credit_card_project.silver.transactions`
- **SCD Type:** Type 2
- **Unique Records Source:** `SELECT DISTINCT customer_id, customer_name, customer_email, customer_phone, customer_address, customer_dob FROM credit_card_project.silver.transactions`

## 2. `dim_card`

This dimension stores details about each unique credit card. It is treated as SCD Type 1, where changes to card attributes (like `card_type`, though unlikely) would overwrite the existing record.

- **Target Table:** `credit_card_project.gold.dim_card`
- **Source:** `credit_card_project.silver.transactions`
- **SCD Type:** Type 1
- **Unique Records Source:** `SELECT DISTINCT card_number, card_type, card_expiration_date FROM credit_card_project.silver.transactions`
